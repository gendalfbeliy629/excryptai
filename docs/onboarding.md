# Onboarding — crypto-AI

## 1. Назначение документа

Документ помогает новому участнику команды быстро понять:
- что это за проект
- как устроен репозиторий
- кто за что отвечает
- как начинать работу
- где искать документацию
- как правильно работать через GitHub

Документ предназначен для новых:
- разработчиков
- системных аналитиков
- тестировщиков
- PM / РП
- тех. писателей

---

## 2. Что такое crypto-AI

crypto-AI — это проект по анализу криптовалютных рынков и генерации торговых сигналов.

Проект включает:
- backend API
- web-приложение
- Telegram-бота
- интеграции с внешними market-data провайдерами
- общую бизнес-логику расчёта сигналов
- документацию и процессы работы команды

## 4. С чего начать новому участнику
### Шаг 1. Прочитать базовые документы
Обязательно:
- README.md
- CONTRIBUTING.md
- docs/architecture/README.md
- docs/api/README.md
- docs/business-rules/README.md

### Шаг 2. Получить доступы
Потребуется доступ к:
- GitHub repository
- GitHub Project
- Wiki
- deployment / logs / environments
- внешним сервисам, если это требуется по роли

### Шаг 3. Настроить окружение
Типовые шаги:
- установить Node.js
- установить pnpm
- склонировать репозиторий
- установить зависимости
- настроить .env
- запустить локально backend / web

Пример:
```bash
pnpm install
pnpm dev
```

### Шаг 4. Посмотреть активные задачи
Открыть:
- GitHub Project
- assigned Issues
- текущий sprint / release scope

### Шаг 5. Понять свой контур ответственности
- что именно входит в твою роль
- какие документы и артефакты ты обновляешь
- кто твой основной reviewer / owner / contact person

## 5. Как устроен процесс работы
Основной flow:
- Создаётся Issue
- Issue попадает в Project
- Выполняется анализ
- Задача берётся в работу
- Создаётся branch
- Делается PR
- Проходит review
- Идёт QA
- Задача закрывается
- Документация обновляется

Ключевое правило:
- никакой "теневой" работы вне GitHub-процесса

## 6. GitHub-артефакты, которые нужно знать
### Issues
Используются для:
- feature
- bug
- task
- docs
- analysis
- tech debt

### Pull Requests
Используются для:
- review
- обсуждения решения
- фиксации изменения
- контроля качества перед merge

### Project
- Используется для управления статусами задач.

### Wiki
Используется как база знаний:
- onboarding
- glossary
- troubleshooting
- process docs

### docs/
- Используется для versioned-документации рядом с кодом.

## 7. Статусы задач в Project
Используются статусы:
- Backlog — задача ещё не подготовлена
- Ready for Analysis — нужна проработка
- Ready for Dev — можно начинать реализацию
- In Progress — задача в работе
- In Review — открыт PR
- Ready for QA — можно тестировать
- In QA — тестирование идёт
- Done — работа подтверждена
- Blocked — задача упёрлась в блокер

## 8. Роли в проекте
### System Analyst
Отвечает за:
- требования
- acceptance criteria
- бизнес-правила
- API-контракты
- системную документацию

### Developer
Отвечает за:
- реализацию
- структуру кода
- интеграции
- code review
- техдолг
- обновление docs при техизменениях

### QA
Отвечает за:
- тестовые сценарии
- bug reports
- проверку логики
- регрессионную проверку

### PM / РП
Отвечает за:
- приоритеты
- планирование
- контроль сроков
- release scope

### Tech Writer
Отвечает за:
- README
- Wiki
- user/admin guides
- release notes
- консистентность документации

## 9. Что важно помнить новичку
- не начинай работу без Issue
- не делай merge без PR
- не меняй бизнес-логику без обновления документации
- не оставляй незафиксированные договорённости "в чате"
- не считай README и docs второстепенными артефактами
