# Архитектура crypto-AI

## 1. Назначение документа

Документ описывает архитектуру проекта crypto-AI на верхнем уровне:
- состав системы
- основные компоненты
- взаимодействие между ними
- внешние интеграции
- ключевые архитектурные решения
- зоны ответственности компонентов

Документ предназначен для:
- системного аналитика
- разработчиков
- тестировщиков
- РП / PM
- тех. писателя
- новых участников команды

---

## 2. Обзор системы

crypto-AI — это система анализа крипторынка, которая объединяет:
- backend API
- frontend web-приложение
- Telegram-бота
- интеграции с внешними источниками рыночных данных
- внутреннюю логику расчёта торговых сигналов
- проектную и техническую документацию

Система предназначена для:
- получения рыночных данных
- обработки и нормализации данных
- расчёта аналитических показателей
- генерации торговых сигналов
- отображения результатов в web-интерфейсе
- отправки сигналов и ответов через Telegram-бота

---

## 3. Архитектурный принцип

Проект строится как **monorepo** с разделением по приложениям и общим пакетам.

Основные принципы:
- разделение frontend и backend
- вынесение общей логики в shared packages
- изоляция внешних интеграций
- минимизация дублирования бизнес-логики
- единый источник правды по правилам анализа
- контролируемое изменение API и бизнес-правил
- документация рядом с кодом

---

## 4. Состав системы

### 4.1. Frontend (`apps/web`)
Отвечает за:
- отображение dashboard
- отображение рынков и карточек инструментов
- визуализацию аналитики
- отображение сигналов
- взаимодействие пользователя с системой через web UI

### 4.2. Backend API (`apps/api`)
Отвечает за:
- предоставление API для frontend
- получение данных из внешних сервисов
- нормализацию и агрегацию данных
- расчёт сигналов
- кеширование
- обработку ошибок интеграций
- подготовку ответов для Telegram-бота

### 4.3. Telegram-бот
Отвечает за:
- обработку пользовательских команд
- выдачу аналитики в текстовом виде
- выдачу сигналов на покупку / продажу
- ответы по рынкам, ценам, сводке и справочной информации

### 4.4. Shared packages (`packages`)
Содержат:
- общие типы
- утилиты
- форматтеры
- общую бизнес-логику
- общие схемы / DTO / validators
- повторно используемые модули

### 4.5. Внешние источники данных
Система может использовать:
- market data providers
- OHLC/candles providers
- sentiment / liquidity / metadata services
- сторонние API криптобирж и аналитических сервисов

## 5. Высокоуровневая схема взаимодействия

```text
Пользователь
   │
   ├── Web UI (apps/web)
   │        │
   │        ▼
   │     Backend API (apps/api)
   │        │
   │        ├── Signal Engine
   │        ├── Market Data Service
   │        ├── Cache Layer
   │        ├── Formatting Layer
   │        └── Integration Layer
   │                 │
   │                 ▼
   │         Внешние market-data сервисы
   │
   └── Telegram Bot
            │
            ▼
        Backend API / shared logic
```

## 6. Основные архитектурные слои
### 6.1. Presentation layer
Слой представления:
- web UI
- Telegram message formatting
- response models для клиента

Задачи:
- показать пользователю результат
- не содержать тяжёлую доменную логику
- не дублировать расчёт сигналов

### 6.2. Application layer
Слой прикладной логики:
- orchestration use cases
- последовательность вызовов сервисов
- построение сценариев работы
- управление потоком выполнения

Примеры сценариев:
- получить список рынков
- собрать сводку сигналов
- рассчитать buy signal
- получить детали по конкретному symbol

### 6.3. Domain / business logic layer
Слой бизнес-логики:
- правила формирования сигналов
- правила фильтрации рынков
- расчёт risk/reward
- расчёт зон входа / stop-loss / take-profit
- правила подтверждения сигналов
- сценарии обработки частично недоступных данных

### 6.4. Integration layer
Слой интеграций:
- клиенты внешних API
- адаптеры данных
- нормализация ответа
- обработка нестабильных ответов
- rate limit handling
- retry / fallback / degrade mode

6.5. Data / cache layer
Слой хранения и кеширования:
- временный кеш рыночных данных
- кеш сигналов
- кеш справочной информации
- механизм инвалидации / TTL

## 7. Основные компоненты backend
### 7.1. API controllers / route handlers
Назначение:
- принимать запросы
- валидировать входные параметры
- передавать выполнение в сервисы
- возвращать ответ в согласованном формате

Не должны:
- содержать тяжёлую бизнес-логику
- напрямую решать правила сигналов

### 7.2. Market Data Service
Назначение:
- обращаться к внешним market-data сервисам
- получать цены, OHLC, объёмы, метаданные
- унифицировать ответы различных провайдеров

### 7.3. Signal Engine
Назначение:
- рассчитывать buy / sell / hold
- анализировать индикаторы и условия входа
- формировать уровни риска
- применять фильтры качества сигнала
- возвращать объяснимый результат

### 7.4. Cache Service
Назначение:
- снижать количество повторных запросов
- ускорять ответы frontend и Telegram-бота
- уменьшать вероятность rate-limit ошибок
- обеспечивать временную устойчивость к нестабильности внешних API

### 7.5. Formatter / Presenter
Назначение:
- формировать понятный ответ для web и Telegram
- приводить данные к UI-friendly формату
- отделять расчёт от представления

### 7.6. Error Handling / Resilience Layer
Назначение:
- не ломать систему целиком при плохом ответе одного провайдера
- возвращать деградированный, но полезный ответ
- логировать причины сбоев
- поддерживать частичную доступность данных

## 8. Основные компоненты frontend
## 8.1. Pages / routes
Примеры страниц:
- dashboard
- markets
- market details
- signal summary
- help / analytics

### 8.2. UI components
Повторно используемые компоненты:
- карточки рынков
- таблицы
- графики
- модальные окна
- индикаторы загрузки
- предупреждения / error banners

### 8.3. API client layer
Назначение:
- централизованный вызов backend API
- унифицированная обработка ошибок
- retry / fallback на UI-уровне, если допустимо
- приведение ответа к состоянию интерфейса

### 8.4. State management
Назначение:
- хранение текущего состояния страницы
- управление кешированием на клиенте
- управление фильтрами, параметрами, результатами загрузки

## 9. Telegram-бот в архитектуре
Telegram-бот является отдельным входным каналом в систему.

Его задачи:
- принять команду пользователя
- вызвать соответствующий сценарий в backend / shared logic
- получить результат анализа
- вернуть ответ в читаемом текстовом формате

Ключевое требование:
- логика сигналов у Telegram-бота и web-приложения должна быть единой
- нельзя дублировать и расходиться в расчётах между каналами

## 10. Внешние интеграции
Внешние провайдеры используются для получения:
- цен
- свечных данных
- объёмов
- ликвидности
- sentiment / metadata
- биржевых сведений

Требования к интеграциям:
- адаптерный слой
- нормализация ответа
- обработка rate limits
- graceful degradation
- логирование ошибок
- таймауты и контроль повторов
- возможность замены провайдера с минимальным влиянием на доменную логику

## 11. Нефункциональные требования к архитектуре
### 11.1. Надёжность
Система не должна полностью падать при:
- ошибке одного внешнего провайдера
- отсутствии части данных
- таймауте одного запроса

### 11.2. Производительность
Система должна:
- использовать кеширование
- не запрашивать одни и те же данные без необходимости
- минимизировать лишние повторные вызовы
- контролировать число запросов к внешним API

### 11.3. Расширяемость
Должна быть возможность:
- добавить нового провайдера
- изменить алгоритм сигналов
- добавить новые страницы frontend
- добавить новые команды бота
- вынести общую логику в shared package

### 11.4. Поддерживаемость
Архитектура должна обеспечивать:
- понятную структуру кода
- разделение ответственности
- предсказуемые точки изменений
- актуальную документацию

### 11.5. Наблюдаемость
Нужно логировать:
- ошибки внешних интеграций
- деградированные режимы
- причины отказа в сигнале
- rate-limit события
- критичные исключения
- ключевые шаги фоновых сценариев

## 12. Основные архитектурные риски
- нестабильность внешних market-data API
- rate limiting у провайдеров
- расхождение бизнес-логики между web и Telegram
- дублирование расчётов в нескольких местах
- рост связности между слоями
- отсутствие единых доменных моделей
- неактуальная документация

## 13. Принятые архитектурные правила
- бизнес-логика расчёта сигналов должна быть централизована
- frontend не должен дублировать расчёт сигналов
- Telegram-бот использует ту же логику, что и web/API
- ответы внешних сервисов должны проходить нормализацию
- ошибки интеграций не должны ломать весь пользовательский сценарий
- документация обновляется вместе с изменением архитектурно значимых частей
