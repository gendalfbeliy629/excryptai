# API crypto-AI

## 1. Назначение документа

Документ описывает API проекта crypto-AI на уровне:
- назначения endpoints
- принципов проектирования API
- правил формата запросов и ответов
- обработки ошибок
- версионирования
- связи API с бизнес-логикой

Документ предназначен для:
- системного аналитика
- backend-разработчиков
- frontend-разработчиков
- QA
- тех. писателя

---

## 2. Общая роль API в системе

Backend API является единым прикладным интерфейсом между:
- frontend-приложением
- Telegram-ботом
- внутренними сервисами и внешними источниками данных

API должно:
- возвращать данные в согласованном формате
- скрывать сложность внешних интеграций
- централизовать бизнес-логику, а не размазывать её по клиентам
- поддерживать стабильные контракты

---

## 3. Основные принципы API

### 3.1. Единый формат ответов
Ответы должны быть предсказуемыми по структуре.

### 3.2. Ясная ошибка вместо молчаливого падения
Если данные недоступны, API должно:
- явно сообщать об этом
- по возможности возвращать частичный результат
- не падать без понятного сообщения

### 3.3. Разделение transport-level и business-level ошибок
Нужно различать:
- техническую ошибку запроса
- бизнес-ситуацию "сигнал не найден"
- частичную доступность данных
- invalid input

### 3.4. Обратная совместимость
Изменения API не должны неожиданно ломать клиентов.

### 3.5. API не должно копировать внешний формат провайдера
Внешние ответы должны нормализоваться во внутреннюю контрактную модель.

---

## 4. Базовые категории endpoints

Ниже — рекомендуемая логическая группировка API.

### 4.1. Service / health endpoints
Примеры:
- health check
- readiness check
- service info

Назначение:
- мониторинг
- деплой
- техническая диагностика

### 4.2. Market endpoints
Примеры:
- список доступных рынков
- информация по инструменту
- текущая цена
- OHLC / historical data
- market overview

### 4.3. Signal endpoints
Примеры:
- расчёт buy signal
- расчёт sell signal
- signal summary
- signal details
- рекомендации по входу / stop-loss / take-profit

### 4.4. Analytics endpoints
Примеры:
- агрегированная аналитика
- summary metrics
- market regime
- объяснение результата анализа

### 4.5. Reference / metadata endpoints
Примеры:
- справочная информация по активу
- поддерживаемые symbols
- конфигурационные данные для UI

---

## 5. Рекомендуемый стиль маршрутов

Нужно использовать читаемые, предметно-ориентированные маршруты.

Примеры:
- `/api/health`
- `/api/markets`
- `/api/markets/{symbol}`
- `/api/markets/{symbol}/history`
- `/api/signals/buy`
- `/api/signals/sell`
- `/api/signals/summary`
- `/api/assets/{symbol}/info`

Рекомендации:
- использовать существительные для ресурсов
- не смешивать разные смысловые области в одном endpoint
- action-like routes использовать только там, где это действительно сценарий расчёта

## 6. Примеры логических контрактов

### 6.1. Health response
Назначение:
- показать, что сервис жив
- вернуть базовое техническое состояние

Пример структуры:
```json
{
  "success": true,
  "service": "crypto-ai-api",
  "status": "ok"
}
```

### 6.2. Market item
Назначение:
- вернуть краткую информацию по инструменту

Пример структуры:
```json
{
  "symbol": "BTC/USDT",
  "base": "BTC",
  "quote": "USDT",
  "price": 68250.12,
  "change24h": 1.42,
  "volume24h": 123456789
}
```

### 6.3. Signal response
Назначение:
- вернуть результат анализа сигнала

Пример структуры:
```json
{
  "symbol": "BTC/USDT",
  "signal": "BUY",
  "confidence": "medium",
  "entryZone": {
    "from": 67000,
    "to": 67500
  },
  "stopLoss": 64800,
  "takeProfits": [
    69000,
    70500,
    72000
  ],
  "explanation": [
    "Подтверждён импульс после ретеста",
    "Объём поддерживает сценарий входа"
  ]
}
```

## 7. Формат успешного ответа
```json
{
  "success": true,
  "data": {},
  "meta": {}
}
```
Где:
- success — технический результат выполнения запроса
- data — полезная нагрузка
- meta — метаданные, если они нужны


## 8. Формат ошибочного ответа
```json
{
  "success": false,
  "error": {
    "code": "MARKET_DATA_UNAVAILABLE",
    "message": "Не удалось получить рыночные данные",
    "details": {}
  }
}
```

Требования к ошибкам:
- code должен быть машинно-читаемым
- message должен быть понятен человеку
- details — необязательный объект для диагностики
- ошибки одного внешнего провайдера не должны без необходимости превращаться в 500 для всего сценария

## 9. Классы ошибок
### 9.1. Validation errors

Примеры:
- некорректный symbol
- отсутствующий обязательный параметр
- неподдерживаемый timeframe

### 9.2. Not found / unsupported
Примеры:
- актив не найден
- symbol не поддерживается
- инструмент недоступен

### 9.3. External dependency errors
Примеры:
- таймаут провайдера
- invalid upstream response
- rate limit
- провайдер недоступен

### 9.4. Business-result errors
Примеры:
- сигнал не найден
- недостаточно данных для анализа
- сценарий подтверждения не выполнен

### 9.5. Internal errors
Примеры:
- непредвиденное исключение
- ошибка сериализации
- ошибка внутренней конфигурации

## 10. HTTP status подход
- 200 — запрос обработан успешно
- 400 — ошибка в запросе клиента
- 404 — ресурс не найден
- 409 — конфликт состояния, если применимо
- 422 — корректный запрос, но невалиден с точки зрения бизнес-правил
- 429 — rate limit
- 500 — внутренняя ошибка сервиса
- 502/503/504 — проблемы внешней зависимости / доступности, если это реально отражает ситуацию

Важно:
- бизнес-сценарий "сигнал отсутствует" не всегда должен быть 500
- частичная деградация лучше, чем полное падение

## 11. Версионирование API
Подход:
- явное versioning, если контракт стабилизирован и поддерживается несколькими клиентами
- например: /api/v1/...

Если версия не введена на раннем этапе:
- изменения контракта фиксируются в changelog и docs
- при росте системы версия должна быть добавлена

## 12. Требования к API-контрактам
Для каждого значимого endpoint должно быть зафиксировано:
- назначение
- входные параметры
- формат ответа
- возможные ошибки
- пример успешного ответа
- пример ошибочного ответа
- ограничения
- влияние на frontend / Telegram-бота

Рекомендуется:
- вести OpenAPI / Swagger
- синхронизировать контракт с frontend и QA
- менять контракт только осознанно

## 13. Требования к интеграционному слою API
API не должно напрямую "пробрасывать" сырой ответ внешнего сервиса.

Нужно:
- нормализовать входящие данные
- валидировать критичные поля
- обрабатывать пустые / повреждённые ответы
- использовать fallback или degrade mode
- логировать причину ошибки

## 14. Требования к observability
Для API должны логироваться:
- входной запрос
- ключевые этапы выполнения use case
- ошибки внешних интеграций
- частичная деградация
- rate limit
- unexpected exceptions

При этом:
- нельзя логировать чувствительные данные без необходимости
- логи должны помогать разбирать инцидент

## 15. Влияние API на frontend и Telegram
Критическое правило:
- web и Telegram не должны иметь две разные бизнес-логики
- API / shared domain logic должны быть единым источником правды

Frontend должен использовать API как контрактный слой, а не самостоятельно восстанавливать бизнес-правила из сырых данных.

## 16. Definition of Ready для API-изменений
Изменение API готово к разработке, если:
- понятна цель endpoint
- описан вход и выход
- определены ошибки
- понятен потребитель (web / bot / internal)
- описано влияние на существующих клиентов

## 17. Definition of Done для API-изменений
Изменение API считается завершённым, если:
- endpoint реализован
- контракт зафиксирован
- обработка ошибок реализована
- QA-сценарии подготовлены
- документация обновлена
- frontend / bot impact учтён
