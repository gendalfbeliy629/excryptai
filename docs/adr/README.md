# ADR — Architectural Decision Records
(Архитектурные решения)

## 1. Назначение документа

В этой папке фиксируются архитектурные решения проекта crypto-AI.

ADR нужен, чтобы:
- помнить, почему было принято то или иное решение
- не спорить заново о том, что уже осознанно решено
- объяснять новым участникам команды контекст изменений
- отделять "случайно так получилось" от "мы это осознанно выбрали"

---

## 2. Когда нужно создавать ADR

ADR создаётся, если решение:
- влияет на архитектуру системы
- влияет на API-контракты
- влияет на способ интеграции с внешними сервисами
- влияет на структуру монорепозитория
- влияет на стратегию кеширования / rate limit / resilience
- затрагивает несколько командных ролей
- будет иметь долгосрочные последствия

Примеры:
- выбор monorepo структуры
- выбор стратегии кеширования сигналов
- решение использовать единый signal engine для web и Telegram
- решение о формате API-ошибок
- решение о fallback / degrade mode при нестабильности провайдеров

---

## 3. Когда ADR не нужен

ADR не нужен для:
- мелкой правки текста
- локального bugfix без архитектурных последствий
- косметического рефакторинга
- незначительного rename

---

## 4. Как именовать ADR

Рекомендуемый формат:
`NNN-short-title.md`

Примеры:
- `001-monorepo-structure.md`
- `002-unified-signal-engine.md`
- `003-market-data-cache-strategy.md`
- `004-api-error-format.md`

Нумерация должна быть последовательной.

---

## 5. Рекомендуемый шаблон ADR

Каждый ADR должен содержать:

### 1. Заголовок
Краткое название решения.

### 2. Статус
Например:
- proposed
- accepted
- superseded
- deprecated

### 3. Контекст
Почему возник вопрос и что нужно решить.

### 4. Проблема
Какую проблему решает ADR.

### 5. Рассмотренные варианты
Какие альтернативы обсуждались.

### 6. Принятое решение
Что именно выбрали.

### 7. Последствия
Плюсы, минусы, ограничения, компромиссы.

### 8. Влияние
На какие компоненты влияет решение.

### 9. Связанные документы
Ссылки на architecture, API, issue, PR, diagrams.

## 7. Правила ведения ADR
- один ADR = одно решение
- ADR должен быть кратким, но содержательным
- ADR должен описывать не только выбор, но и причину
- ADR нельзя писать задним числом без контекста
- если решение изменилось, создаётся новый ADR, а старый помечается как superseded

## 8. Кто участвует в ADR
## System Analyst
- фиксирует бизнесовый и системный контекст
- следит за ясностью формулировок
- помогает связать решение с требованиями

## Developer / Tech Lead
- описывает технические варианты
- оценивает последствия
- отвечает за реализуемость решения

## QA
- помогает выявить риски, тестируемость и влияние на сценарии

## PM / РП
- оценивает влияние на сроки, scope, приоритет

## Tech Writer
- помогает привести решение к понятному документируемому виду

## 9. Примеры потенциальных ADR для crypto-AI
- monorepo как базовая структура проекта
- единый signal engine для всех каналов
- стратегия обработки upstream ошибок
- обязательный degrade mode вместо полного падения
- кеширование рыночных данных с TTL
- staged scan / rate limit strategy
- стандартизация формата ответа API
- вынесение бизнес-логики в shared packages
