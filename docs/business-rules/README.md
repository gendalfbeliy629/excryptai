# Бизнес-правила crypto-AI

## 1. Назначение документа

Документ фиксирует бизнес-правила и доменные правила проекта crypto-AI.

Цель документа:
- отделить бизнес-логику от технической реализации
- зафиксировать единые правила для web, API и Telegram-бота
- уменьшить риск расхождения логики между компонентами
- дать основу для анализа, разработки и тестирования

Документ предназначен для:
- системного аналитика
- backend-разработчиков
- frontend-разработчиков
- QA
- PM
- тех. писателя

---

## 2. Домен проекта

Основной домен проекта:
- анализ криптовалютных рынков
- оценка торговых условий
- генерация торговых сигналов
- объяснимое представление результата пользователю

Система не должна вести себя как "магическая коробка".
Каждое решение по сигналу должно быть:
- объяснимым
- воспроизводимым
- основанным на данных
- ограниченным правилами риска

---

## 3. Ключевые сущности домена

### 3.1. Инструмент / рынок
Сущность торговой пары, например:
- BTC/USDT
- ETH/USDT
- ONT/USDT

Ключевые атрибуты:
- base asset
- quote asset
- текущая цена
- исторические данные
- объёмы
- доступность данных

### 3.2. Торговый сигнал
Результат анализа рынка.
Допустимые значения:
- BUY
- SELL
- HOLD

Сигнал должен быть:
- результатом набора правил
- объяснён пользователю
- привязан к конкретному symbol и моменту времени

### 3.3. Зона входа
Ценовой диапазон, в котором вход считается допустимым.

### 3.4. Stop-loss
Уровень ограничения риска.

### 3.5. Take-profit
Целевые уровни фиксации прибыли.

### 3.6. Подтверждение сигнала
Дополнительное условие, необходимое для повышения качества входа.
Например:
- закрытие свечи
- ретест уровня
- breakout-retest сценарий

---

## 4. Главные бизнес-принципы

### 4.1. Единая логика сигнала
Один и тот же рынок при одинаковых входных данных должен давать одинаковый сигнал:
- в API
- в web
- в Telegram-боте

### 4.2. Нет сигнала без достаточных данных
Если данных недостаточно:
- нельзя симулировать уверенность
- нужно вернуть понятный результат: недостаточно данных / сигнал не подтверждён / рынок пропущен

### 4.3. Риск-менеджмент обязателен
Если система рекомендует BUY или SELL, она должна при возможности возвращать:
- зону входа
- stop-loss
- take-profit
- оценку риска / комментарий по управлению позицией

### 4.4. Сигнал должен быть объяснимым
Система должна уметь ответить:
- почему сигнал BUY
- почему сигнал HOLD
- почему рынок был отброшен
- чего не хватило для подтверждения

### 4.5. Частичная деградация лучше полного отказа
Если часть рынков недоступна или часть провайдеров не ответила:
- система по возможности продолжает работу
- пользователю сообщается, что часть данных недоступна

---

## 5. Базовая логика сигналов

### 5.1. BUY
BUY допустим только если:
- данные по рынку доступны
- выполнены базовые условия стратегии
- вход не противоречит риск-ограничениям
- сигнал не признан шумовым
- при необходимости выполнено подтверждение входа

### 5.2. SELL
SELL допустим, если:
- выполнены условия выхода
- достигнуты условия фиксации
- нарушены условия удержания позиции
- стратегия явно указывает на выход

### 5.3. HOLD
HOLD возвращается, если:
- нет качественного BUY
- нет обоснованного SELL
- рынок не даёт подтверждения
- данные частично доступны, но недостаточны для действия

---

## 6. Правила качества сигнала

Система не должна возвращать сигнал только потому, что "что-то похоже".

Сигнал должен фильтроваться по качеству.

Факторы качества:
- полнота данных
- согласованность индикаторов
- наличие подтверждения
- отсутствие критичных конфликтов между условиями
- допустимый риск / reward
- рыночный контекст

---

## 7. Правила подтверждения сигнала

Для повышения качества входа система может использовать дополнительный слой подтверждения.

Примеры:
- вход только после закрытия 1H candle
- вход после breakout-retest
- вход после подтверждения удержания уровня
- вход после возврата цены в допустимую зону

Если подтверждение обязательно, но не получено:
- сигнал не должен маркироваться как полноценный BUY
- нужно вернуть HOLD или "ожидается подтверждение"

---

## 8. Правила обработки рынков

### 8.1. Проверка рынка
Каждый рынок анализируется отдельно.

### 8.2. Пропуск рынка
Рынок может быть пропущен, если:
- не хватает данных
- данные повреждены
- рынок не поддерживается
- внешний провайдер не ответил
- инструмент не соответствует критериям отбора

### 8.3. Отчёт по пропущенным рынкам
Если сценарий предполагает анализ набора рынков, система должна по возможности сообщать:
- какие рынки не удалось проверить
- почему не удалось проверить

---

## 9. Правила формирования buy summary

Если сценарий возвращает подборку рынков для покупки:
- в выдачу должны попадать только рынки, прошедшие правила отбора
- список должен быть отсортирован по выбранному критерию
- причины попадания должны быть объяснимы
- неподтверждённые кандидаты не должны смешиваться с подтверждёнными сигналами без пометки

Если нет рынков, удовлетворяющих условиям:
- система должна явно сообщить, что качественных BUY-сигналов нет
- при возможности должна кратко объяснить причину

---

## 10. Правила risk management

Минимум для сигнала:
- известна точка или зона входа
- есть stop-loss
- есть минимум одна цель
- риск не должен быть неограниченным

Желательно:
- несколько take-profit уровней
- логика break-even
- пояснение, когда переносить stop-loss
- пояснение по подтверждению before entry

---

## 11. Правила объяснения результата пользователю

Пользователь должен понимать:
- что делать
- на каком основании это предложено
- какие есть риски
- что делать, если подтверждение ещё не произошло

Формат объяснения должен быть:
- кратким, но не пустым
- понятным
- без ложной уверенности
- без противоречий между разделами ответа

---

## 12. Правила консистентности

Обязательное правило:
- логика расчёта и интерпретации сигнала едина для всех каналов
- нельзя, чтобы web показывал HOLD, а bot — BUY при тех же входных данных, если это не объясняется разницей сценария

Если сценарии различаются:
- различие должно быть явным и документированным

---

## 13. Правила обработки ошибок на уровне бизнеса

Не каждая ошибка является критическим падением.

Допустимые сценарии:
- "данные по рынку временно недоступны"
- "недостаточно данных для анализа"
- "сигнал не подтверждён"
- "часть рынков пропущена"

Эти сценарии должны быть:
- предсказуемы
- описаны
- тестируемы

---

## 14. Что считается нарушением бизнес-правил

- BUY без понятной зоны входа
- BUY без контроля риска
- противоречие между web и Telegram
- отсутствие объяснения, почему сигнал выдан
- молчаливый пропуск ошибки
- смешивание подтверждённых и неподтверждённых сигналов без пометки
- рекомендация действия при недостатке данных

---

## 15. Definition of Ready для изменения бизнес-логики

Изменение бизнес-правила готово к разработке, если:
- понятна цель изменения
- описан текущий и целевой сценарий
- зафиксированы acceptance criteria
- описаны edge cases
- понятны последствия для API / web / bot

---

## 16. Definition of Done для изменения бизнес-логики

Изменение считается завершённым, если:
- правило реализовано
- правило описано в документации
- подготовлены тестовые сценарии
- исключены противоречия между каналами
- объяснение результата обновлено
- QA подтвердил ожидаемое поведение
